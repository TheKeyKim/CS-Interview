> DML이란 무엇이며 어떤 명령어가 있는가?
- 데이터 조작어(Data Manipulation Language)
- 데이터베이스에 들어있는 데이터를 조회, 검색, 변형을 위한 명령어들.
- select, insert, update, delete

> DDL이란 무엇이며 어떤 명령어가 있는가?
- 데이터 정의어(Data Definition Language)
- 데이터 구조를 정의하는데 사용하는 명령어들. 
- 테이블을 만들기도 하고, 구조를 바꾸는 명령어들.
- create, alter, drop, rename, trucate

> DCL이란 무엇이며 어떤 명령어가 있는가?
- 데이터 제어어(Data Control Language)
- 데이터베이스 접근 및 객체들에 대한 사용 권한을 부여 및 박탈하는 명령어들
- Grant, Revoke

> RDBMS의 특징은 무엇이며 어떤 장점이 있는가?
- RDBMS는 관계형 모델을 기반으로하는 DBMS의 유형
- 장점
    - 테이블 간에 연관이 존재하여 DBMS보다 효율적으로 데이터를 저장 및 구성, 관리 가능.
    - 데이터의 원자성, 일관성, 내구성 및 데이터 무결성을 높임
- 단점
    - 시스템 관계의 복잡도를 고려하여 구조화해야함. 
    - 복잡하게 얽힐수록 query문이 복잡해짐.
- MySQL, Oracle, etc..

> NoSql의 특징은 무엇이며 어떤 장점이 있는가?
- Not Only SQL, Non relational Database
- 스키마와 같은 관계에 대한 개념이 존재하지 않음
- 더 자유로운 관리가 가능함. 
- 컬렉션이라는 형태로 테이블과 비슷한 역할을 함.(즉, 컬럼을 몇 개만 골라 테이블 처럼 사용할 수 있음.)
- 장점
    - 스키마가 정해져있지 않아 복잡한 형태를 띨 필요가 없음
    - join과 같은 복잡한 구조를 신경 쓰지 않아도 됨.
    - 프레임워크 자체에 분산처리 기능을 포함한 경우가 많음
- 단점
    - 자유로움 때문에 중복된 데이터가 저장이 가능함.
    - 관리가 어려울 수 있음.
    - key에 의한 put/get만 지원함.(내부적으로 hash map의 형태로 구현되어 있다고 함.)

> B-Tree
- 밸런스드 트리(Blanaced Tree)
- 높이가 과도하게 커지는 것을 방지한 트리임.
- Tree를 정의할 때 max degree를 정의하고, 그에 맞추어 트리를 빌딩함.
- 각 노드는 데이터를 가지고 있으며, n개의 데이터를 가지고 있다면 n+1의 자식노드를 갖음. 

> B-Tree가 DB에 사용되는 이유
- 균형 트리는 여러 형태가 존재함. 그 중에서도 B-Tree가 RB트리와 같은 타 균형 트리보다 우월한 점은 각 노드에 여러 데이터가 들어간다는 것임.
- 포인터로 인한 접근보다 여러 데이터가 배열식으로 들어있는 것은 캐시 친화적이며, 조금 더 빠른 접근이 가능함을 의미함.

> 각종 키는 어떤 것이 있으며, 어떤 특징을 갖는가?
- 기본키(Primary Key)
    - 각 데이터 row가 고유적으로 갖을 수 있는 키의 쌍들의 집합. 
    - '최소'의 key들로 '유일'하게 각 row들을 표현할 수 있어야 한다. 
- 후보키(Candidate Key)
    - Primary Key로 사용될 여지가 있는 키들의 집합
    - 즉, 최소성과 유일성을 둘 다 갖춘 키의 집합
    - (기본키가 하나의 키가 아닌 여러 키로 이루어질 수 있는 만큼 후보키도 마찬가지)
- 대체키(Alternate Key)
    - 후보키 = 기본키 + 대체키
    - 즉 기본키 대신 쓸 수 있는 조건을 만족하는 키들.
- 슈퍼키(Super Key)
    - 유일성은 만족하지만, 최소성을 만족치 못하는 키.
- 외래키(Foreign Key)
    - 다른 릴레이션의 기본키를 참조하는 키. 

> 트랜잭션이란 무엇이며, 어떤 특징을 갖는가? 어떤 조건을 만족하는가?
- 트랜잭션이란, 데이터베이스의 상태를 변환시키는 '일련의 명령어' 집합이다.
- 원자성
    - 트랜잭션은 일련의 명령어 과정이 '모두 정상적으로' 반영되거나, 아예 반영되지 않아야 한다.
    - 즉, 한 연산이라도 잘못되었을시, 모든 연산이 취소되어야 한다.
- 일관성
    - 같은 상태에서 같은 트랜잭션을 실행했을 떄는 늘 일관된 데이터베이스 상태로 변환된다.
- 독립성
    - 둘 이상의 트랜잭션이 병행되는 경우, 서로는 서로의 연산에 끼어들 수 없다.
    - 즉, 하나의 수행중인 트랜잭션이 완료될 때까지 다른 트랜잭션에서 해당 수행결과를 참조할 수 없다.
- 지속성
    - 성공적으로 완료된 트랜잭션은, 시스템에 영구적으로 반영되어야 한다.
- 트랜잭션은 commit 과 rollback의 두 연산을 갖는다
    - commit : 일련의 명령어를 모두 실행시킨다.
    - rollback : 원자성을 저해할 요소가 발생할 시, 해당 트랜잭션에서 행해졌던 모든 결과들을 취소시킨다.

> DB의 데이터 무결성과 정합성이란 무엇인가?
- 무결성(Integrity) 
    - 데이터 값 자체가 올바른지 여부
- 정합성(Consistency)
    - 데이터들의 값이 서로 일치함
    - 중복되는 데이터가 많이 사용될 경우에, 정합성이 깨지기 쉬움
    - 데이터의 모순이 없어야함을 의미함.

- 정합성은 있으나 무결성이 깨진 경우
    - 중복된 데이터의 값이 모두 같지만, 애당초 그 값이 옳지 않은 경우
    - ex) '홍길동'의 나이는 모든 튜플에서 24세로 일치하지만, 실제 나이는 28세라 데이터가 올바르지 않음
- 정합성이 깨지면, 무결성도 깨짐

> 무결성의 종류는? 그리고 각각의 설명?
- 개체 무결성(Entity Integrity)
    - 릴레이션에서 PK의 속성(Attribute)들은 NULL이나 중복값을 가질 수 없음
    - NOT NULL, UNIQUE, PK
- 참조 무결성(Referential Integrity)
    - FK(외래키) 값은 NULL이거나 참조 릴레이션의 값과 동일해야함
    - FK
- 도메인 무결성(DOMAIN Integrity)
    - 같은 속성에 사용된 값들은 같은 성격의 값을 갖음
    - 같은 영역의 값을 갖는다는 것
    - 성별 속성의 도메인은 [남, 여] 로 정해져있음.
    - NOT NULL, ENUM data type
- (추가) 업무 무결성
    - 기업 혹은 단체에서 업무를 수행할 때 제약을 걸어둠
    - 예를 들어, 특정 물건의 최저가는 1000원 이하가 되어서는 안된다는 조건이 있을 수 있음.

> 이상현상(Anomaly)란 무엇인가?
- 데이터베이스에서 일어나는 이상현상들
- 갱신 이상(Modification Anomaly)
    - 반복된 데이터 중 일부를 갱신할 때 데이터가 불일치
    - 정합성을 깨치는 행위
    - 예를 들어, 학사시스템의 col에 '학부'가 속해있고, 같은 사람을 지칭하는 tuple이 중복되게 존재 할 때, 특정 학생이 전과를 신청할 경우, 중복된 데이터를 모두 수정해야 한다. 이때, 제대로 이뤄지지 않는다면, 갱신이상이 발생했다고 볼 수 있다.
- 삽입 이상(Insertion Anomaly)
    - 새로운 데이터를 추가할 때 불필요한 데이터도 함께 삽입해야하는 경우를 의미함
    - 예를 들어, 학사 시스템의 col에 '졸업학점'이 추가 될 경우 재학생들에게는 해당 되지 않는 column이므로, 불필요한 값이 들어가게 된다.
- 삭제 이상(Deletion Anomaly)
    - 튜플을 삭제할 때 필요한 데이터까지 함께 삭제되는 데이터 손실의 경우

> 정규화란 무엇이고 왜 필요한가?
- 정규화란 이상현상(Anormaly)를 막기 위하여 스키마를 리팩토링하는 것이라 볼 수 있다
- 데이터 저장을 논리적, 직관적으로 하기 위함이다.
1. 1차 정규화
    - 각 로우마다 컬럼의 값의 갯수 제한을 1개로 한다.
    - 즉 데이터는 배열의 형태를 띄지 않는다.
    - 여러 값이 필요할 경우에는, 다른 튜플을 만든다.
    - 데이터 Redundancy는 증가하였으나, 데이터의 논리적 구성을 위한 trade-off다.
2. 2차 정규화
    - 테이블의 모든 컬럼이 완전 함수적 종속을 만족해야 한다.
    - 그 말인 즉슨, pk가 여러개일 때 전체 pk가 아닌 특정 attribute에 따라서 해당 값이 정해져서는 안된다는 말임
    - **예시**
    ```
    name    age     favorite_food
    -----------------------------
    홍길동    28        송편
    홍길동    28        죽엽청
    고길동    44        광어회
    최둘리    11        아이스크림
    최둘리    11        막걸리
    ```
    - 위의 경우 Age는 PK(name, favorite_food)에서 name에만 부분 종속적인 특성이므로 완전 함수적 종속을 만족시키지 않는다. 그러므로 이를 다른 데이터 릴레이션으로 분리해야만 할 것이다.

3. 3차 정규화
    - 이행적 함수 종속이 없어야 한다
    - 이행적 함수 종속이란, pk를 제외한 다른 특성이 다른 Attribute를 결정하는 일이 없어야 한다.
    - **예시**
    ```
    person_id  | name  |  city   |  street  |  Zip_code
    ```
    - 위와 같은 컬럼들이 있을 때 zip_code(우편번호)는 주민의 pk는 되지 못하지만, 다른 특성(Attribute)들을 결정한다.
    - 위와 같은 이행적 함수 종속을 없애는 것이 3차 정규화이다.
    - **예시**
    ```
    person_id  | name  |  Zip_code
    -------------------------------
    Zip_code  |  city   |  street  
    ```
    - 위와 같이 릴레이션을 나누어 주어야 한다.

4. BCNF
    - Boyce and Codd Normal Form
    - 일반 컬럼이 후보키를 결정해서는 안된다.
    - (추가 필요)

> 뷰란 무엇이고 어떤 장단점을 갖는가?
- View
- 데이터를 보여주기 위하여 만들어진 '가상의 테이블' 
- 실제 테이블의 일부분만을 보여준다(join테이블의 일부분도 가능하다)
- 장점
    - 논리적 독립성을 제어
    - 뷰를 통한 접근은, 다른 특성들을 건드리지 않게 할 수 있다.
    - 여러 뷰를 정의할 수 있다.
- 단점
    - 독자적 인덱스를 갖지 않는다
    - 정의를 변경할 수 없다.
    - 조작에 많은 어려움을 갖는다. 

> join의 종류와 설명?
- inner join
    - 두 테이블에 공통적 'value'만을 이용하여 join한다.
- outer join
    - 한 테이블에만 있어도 join이 가능하다
    - left, right, full
    - 예시) left : 왼쪽 테이블은 겹치는 값이 존재하든, 아니든 모두 가져오고, 오른쪽은 그렇게 하지 않는다.
- natural join  
    - 같은 이름을 갖는 column에 대해 join을 진행한다.
    - 같은 이름을 갖는 column이 데이터 타입이 다를 경우 오류가 발생한다.
